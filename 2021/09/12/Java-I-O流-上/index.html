字节流与字符流

想要更好的理解 I/O 流，计算机的各种操作码及相关的编码知识是必要的。Java 中 I/O 流类可以分为字节流和字符流，实际上，字符流也是在字节流的基础上进行的包装，对于计算机来说，他只认识二进制数字，字符流什么的他是不懂的，所以对于 I/O 流来说，到了最底层还是一个一个的字节

字节是很“固定”的，1字节(1 Byte)就是8位二进制数字(1 Byte=8 bits)，而二进制数字是计算机可以识别的。字符就不那么“固定”了，这与字符的编码方式有关，比如 gbk 编码的中文字符占用两个字节，英文占用一个字节；而 utf-8 编码的中文字符占用三个字节，英文字符占用一个字节。正是由于采用的编码方式不同，使用中可能会出现乱码问题。为了方便我们操作字符，从而在字节的基础上，根据相应的编码方式提供了字符流。字节流和字符流的区别：

  • 读写单位不同：字节流以字节（8bit）为单位，字符流以字符为单位，根据不同的字符编码方式，一次可能读取多个字节
  • 处理对象不同：字节流能处理所有类型的数据（如图片、avi等），而字符流只能处理字符类型的数据

所以，只要是处理纯文本数据，就可以优先考虑使用字符流，除此之外都使用字节流

文件相关的概念

区分几个概念：

  • 文件：字节的序列。如果字节的内容被解释为字符，则文件被称为文本文件，否则文件被称为二进制文件
  • 文本文件：字符序列按照某种编码方式(utf-8，gbk 等) 序列化为 byte 的存储结果
  • Java 文本: 采用 utf-16be 双字节编码，每个中英文字符均占用两个字节(utf-16be：Unicode Transformation Format 16-bit Big Endian: Big Endian即大端存储，高位字节存储在低地址，符合我们的操作习惯)

Java File 类

开始 I/O 流类的总结前，关于 File 类需要提一下。一言以蔽之，File 类仅仅是定义了一个文件的抽象路径名，至于该抽象路径被操作系统解析为路径字符串并进行查找后，文件到底存不存在，无从确定，可能有也可能没有

在Thinking in Java中，关于 File 类的定义我觉得很清晰，引用如下:

The File has a deceiving name; you might think it refers to a file, but it doesn’t. In fact, “FilePath” would have been a better name for the class. It represent either the name of a particular file or the name of a set of files in a directory.

因此，File 类只用于表示文件(目录)的信息(名称，大小等)，不能用于文件内容的访问(RandomAccessFile 类是 Java 提供的文件内容的访问类，可以读写文件，支持随机访问，可实现多线程下载，文件组装等)

这里还有两个重要概念需要拿出来说明一下：

  • 绝对路径与相对路径

绝对路径没什么好说的，绝对路径名是完整的路径名，不需要其他信息就可以定位自身表示的文件；

相对路径：相对路径名必须使用来自其他路径名的信息进行解释，默认情况下，java.io 包中的类总是根据当前用户工作目录来分析相对路径名。此目录由系统属性 user.dir 指定，通常是我们调用 JVM 的路径。在 web 应用中也经常涉及到相对路径的使用，一定要搞清楚相对路径相对的是哪个文件或路径

  • 抽象路径与路径名字符串

File 类是文件或者目录(目录本质也是文件)的抽象路径表示，而操作系统需要使用依赖于特定操作系统的路径名字符串来表示文件和目录；所以这里就涉及两者之间的一个转换，这种转换与不同的操作系统有关；比如，在 windows 操作系统下，以下代码均可以使用：
