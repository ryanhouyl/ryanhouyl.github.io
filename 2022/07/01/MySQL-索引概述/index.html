为什么要使用索引

使用索引的目的是为了加快查询速度，那么为什么索引可以加快查询速度呢？我们知道，一个好的算法一般是建立在一个合理的数据结构的基础上，比如我们这里的谈搜索算法，二分法需要数据是有序的，二叉树查找需要数据满足二叉树的结构，但是很明显，数据库中的数据不可能是有序的或者树形结构的，那么我们怎么办呢？这就有了索引

在数据库存储的数据外建立新的适合查询的数据结构（B+树），也就是索引，通过索引来进行查询，索引的数据项中含有指向具体存储数据的指针，通过该指针可以拿到数据记录，所以本质上来说，索引是一个空间换取时间的设计

B+树

在MySQL中，索引可以hash数据结构，也可以是树形结构，这里我们讨论B+树的数据结构，那么什么是B+树？

1.有n棵子树的结点中含有n个关键字，每个关键字不保存数据，只用来索引，所有数据都保存在叶子节点。

2.所有的叶子结点中包含了全部关键字的信息，及指向含这些关键字记录的指针，且叶子结点本身依关键字的大小自小而大顺序链接。

3.所有的非终端结点可以看成是索引部分，结点中仅含其子树（根结点）中的最大（或最小）关键字。

根据上图结构，假设我们需要查找值为28的数据，只需要三次磁盘IO和三次内存操作，其中内存操作相对磁盘操作可以忽略不计(磁盘IO的执行时间比内存操作差几个数量级)。所以我们也可以说，数据库的索引是为了尽可能的减少磁盘IO(三层的B+树结构可以表示上百万的数据)

根据B+树的结构，我们可以知道磁盘的访问次数等于树高，树高的计算公式为：h = log(m+1)N；其中 N 为数据总量，m 为每个磁盘块中的数据量，m = 磁盘块的大小/数据项的大小；其中数据总量 N 和磁盘块的大小都是固定的，所以要想树高变小，那么应该保持数据项变小，也就是我们在建立索引时，索引列的数据类型尽可能的占用较小的存储

索引的使用

索引的建立和维护是由数据库来完成的，但是理解索引的结构和原理对于理解索引为什么能加快查找速度和如何更好的建立索引是很有帮助的

既然索引是用空间换取时间，所以我们应该在建立索引时考虑空间的问题，比如，尽可能的扩展现有索引而不是建立新的索引；索引是B+树结构，数据库需要维护树形结构，所以需要考虑数据频繁增删改对于索引更新的影响

联合索引

  • 创建一个联合索引的作用相当于创建了多个索引，比如创建(a,b,c)的联合索引相当于建立(a),(a,b),(a,b,c)三个索引
  • 通过联合索引来实现索引覆盖，这样查询到索引数据后就可以直接返回，不需要根据数据项的指针来查询具体的数据记录，减少了随机IO
  • 索引的列越多，通过索引筛选的数据就会越少
  • 联合索引的最左前缀匹配：MySQL会一直向右匹配直到遇到范围查询**(>、<、between、like)就停止匹配**，比如a = 1 and b = 2 and c > 3 and d = 4 如果建立(a,b,c,d)顺序的索引，d是用不到索引的，如果建立(a,b,d,c)的索引则都可以用到，a,b,d的顺序可以任意调整
  • =和in可以乱序，比如a = 1 and b = 2 and c = 3 建立(a,b,c)索引可以任意顺序，mysql的查询优化器会帮你优化成索引可以识别的形式

索引列不要参与运算

  • 比如from_unixtime(create_time) = ’2014-05-29’就不能使用到索引，原因很简单，b+树中存的都是数据表中的字段值，但进行检索时，需要把所有元素都应用函数才能比较，显然成本太大。所以语句应该写成create_time = unix_timestamp(’2014-05-29’)

尽量选择区分度高的列作为索引

  • 区分度的公式是count(distinct col)/count(*)，表示字段不重复的比例，比例越大我们扫描的记录数越少，唯一键的区分度是1，而一些状态、性别字段可能在大数据面前区分度就是0
