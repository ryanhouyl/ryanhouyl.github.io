是什么

浏览器的缓存机制其实主要就是 HTTP 协议的缓存机制(比如 response 头信息中的 Expires, Cache-Control 等字段信息)，也包括部分 HTML 的 Meta 标签，不过说白了，HTML 的 Mete 标签也是通过模拟 HTTP 协议来进行浏览器的缓存控制，如下 HTML 页面代码：

<META HTTP-EQUIV="Pragma" CONTENT="no-cache">

就是告诉浏览器，不使用缓存，直接去服务器请求内容，但是这种方式存在问题，有的浏览器不支持，而且缓存代理服务器由于不能解析 HTML 内容本身，所以不能使用

为什么

为什么使用缓存是一个很显然的问题了，合理的使用缓存可以减轻冗余的数据传输；可以减轻服务器端的负担；使用缓存也可以加快客户端资源的加载；但是既然是缓存就存在一个问题，如果处理不当，会导致页面不能拿到最新的内容进行展示

HTTP协议缓存的处理过程

  • 浏览器第一次请求服务器

浏览器中没有缓存文件，直接请求服务器，服务器返回 200OK ，缓存该次请求的资源(包括 js/css/图片等静态资源)，并设置与缓存相关的头信息，过程示意图如下：

  • 浏览器第 N 次访问服务器(N>1)

浏览器先判断本地的缓存是否过期，用来决定是否要向服务器端发送请求；如果过期，则向服务器端发送请求，这时还需要判断缓存的内容是否更新，如果没有更新，则服务器返回 304 Not Modified，浏览器继续使用本地缓存，否则服务器重新返回请求的数据，过程示意图如下：

HTTP 协议缓存相关的头信息字段

HTTP 响应

在 HTTP 响应头字段中，与缓存有关的是Expires，Cache-Control，Last-Modified和ETag

其中 Expires 和 Cache-Control 字段是服务器控制浏览器是否缓存和缓存时间的字段，而 Expires 在 HTTP/1.0 中已经定义，Cache-Control:max-age=xxx(单位是秒) 在HTTP/1.1中才有定义，所以为了向下兼容，一般会一块儿使用；Expires值为-1和0不缓存。Cache-Control的值为no-cache不缓存.

而 Last-Modified 和 Etag 字段是服务器文件的最后修改时间，浏览器缓存过期时，需要使用该字段来向服务器端进行文件内容的验证，以判断文件自从上次访问后是否有过更新

其中 ETag 是通过哈希算法，根据响应内容生成的实体标记 Entity Tag，该字段的出现是为了解决 Last-Modified 比较难解决的问题：

  • Last-Modified 标注的最后修改时间只能精确到秒级，对于一秒内修改多次的文件无能为力
  • 使用时间可能并不靠谱，比如服务器获取的时间与代理服务器不一致
  • 对于定期生成的文件，内容没有变化，但是 Last-Modified 改变了导致无法使用缓存

HTTP 请求

浏览器先要判断本地缓存是否过期，决定是否要向服务器发送请求；在浏览器请求的头字段中，与缓存有关的信息是If-Modified-Since 和 If-None-Match

If-Modified-Since的值为缓存中最初收到的Last-Modified的值；If-None-Match的值为缓存中最初收到的ETag的值。如果最初的返回中没有Last-Modified和ETag的值则不会带上这个请求参数。服务器收到后会验证，如果相同则返回一个304 Not Modified响应，不会将内容再传一次，浏览器使用本地缓存文件，节省了带宽。

与缓存相关的一些操作

  • 只有 get 请求会被缓存，post 请求不会
  • 当资源分布在不同机器时，同一个资源在不同的机器会生成不同的 ETag，导致 304协议缓存失效，这时就需要自己修改 ETag 的生成方式
  • 系统上线后，在更新资源时，可以在资源 URI 后边附上资源修改时间，svn 版本号，文件 md5 等信息，这样可以避免用户拿到缓存的旧文件
  • F5 刷新浏览器会跳过缓存过期验证，直接去服务器验证缓存内容／Ctrl+F5 强制刷新浏览器会通过 Cache-Control:no-cache 或者 Pragma:no-cache（HTTP 1.0协议中定义)，明确的告诉 Web 服务器浏览器不使用缓存，所以 Web 服务器总会把最新的文件发送给浏览器
