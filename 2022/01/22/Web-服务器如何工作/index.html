你是否考虑过 web 容器或 web 服务器(比如 Tomcat 和 Jboss)是怎么工作的呢？他们是怎么处理来自世界各地的 http 请求呢？这一切的背后到底发生了什么？而 Java 中的 Servlet 提供的 API(也就是 ServletContext，ServletRequest，ServletResponse，Session 类)在其中发挥了怎样的作用？如果你想成为一个 web 开发者，那么这就是不可回避的问题，下面将对这些内容做出相应的回答

区分概念：什么是 web 服务器，应用服务器，web 容器？

先说 web 服务器和应用服务器，一言以蔽之，“在历史中这是两个不同的概念，但是现在这两种不同的范畴慢慢的趋于融合了，现在在大部分的案例和使用中，这两个范畴被视为同一个东西”。下面来说一下服务器的历史：

还在 Mosaic 浏览器(最早的图形化浏览器)和超链接内容的时代，开始诞生了 web 服务器的概念，主要作用是通过 http 协议来处理静态页面内容和图像，所以 so easy，在那时大部分的内容都是静态的， http 1.0 协议仅仅是作为文件传输的工具。不久 web 服务器发展出了 CGI (公共网关接口)功能，这意味 web 服务器可以动态的来处理 web 请求了。与此同时，http 协议也日趋成熟，web 服务器变的越来越强，增加了诸如缓存，安全和会话管理等一系列功能。随着技术的发展，我们有了专门服务于企业的基于 Java 语言的服务器端编程技术，并最终发展为 JSP 技术，一直用到现在

现在我们再来聊聊应用服务器，和 web 服务器类似，应用服务器也是经过了长时间的发展。起初一些公司为 Unix 写了类似 Tuxedo，TopEnd 等类似于 IMS 和 CICS 的主机应用管理和环境监测的程序，而这些程序大都是使用专门的保密的通信协议来互联胖客户机(fat client，也就是 C/S架构的客户端)和服务器。到了 90 年代，这些传统的应用服务器产品开始嵌入 http 通信功能，刚开始时通过网关实现，后来这种界限变得越来越模糊；同时，web 服务器也变得越来越成熟，拥有了处理高负载，高并发等更好的性能；应用服务器也开始添加越来越多的基于 http 协议的功能。所有这一切都使得 web 服务器与应用服务器的界限变得更窄了。虽然应用服务器和 web 服务器间的界限变得越来越窄，但人们还是习惯分开来称呼，主要是为了强调两种服务主要的应用场景

当说 web 服务器时，更多是在强调以 http 协议为核心，以 web UI 为导向的应用；而应用服务器更多的是在于强调高负载，企业级特性，事务和队列，多通道通信(基于 http 协议和其他的协议)。但是现在提供这些需求的基本上都是同一产品

现在我们再来讲一下第三个概念，web 容器

在 Java 中， web 容器也就是 servlet 容器。servlet 容器是用来与 Java Servlets 进行交互的 web 服务器组件。web 容器(servlet 容器)的作用是用来管理 servlet 的生命周期，映射 URL 地址到特定的 servlet，确保 URL 请求拥有正确的访问权限和类似的其他服务。概括之，servlet 容器就是用来运行你的 servlet 并维护它的生命周期的一个运行环境

什么是 Servlet，有啥用？

在 Java 中，Servlet 使我们能够编写基于请求来产生动态内容的服务器端组件。事实上， Servlet 是 javax.servlet 包中定义的一个接口，该接口中定义了三个基本的与 servlet 生命周期相关的方法：init(),service(),destroy()。每个具体的 servlet 类均需要实现这三个方法(要么在 SDK 中，要么自己定义实现)，这三个方法在 servlet 特定的生命周期中被调用

Servlet 类通过类加载器被自动加载到 Servlet 容器，加载的方式分为懒加载和预加载。每一个请求都是一个独立的线程，一个 servlet 类可以同时处理多个线程，当 servlet 对象不再被使用时，他们被 JVM 当做垃圾进行回

什么是 ServletContext，谁创造了它？

当 servlet 容器启动时，它会部署并加载所有的 web 应用，当一个 web 应用被加载时，servlet 容器将会为每一个 web 应用创建一个且仅创建一个 ServletContext 对象，并存储在服务器内存中。同时 web 应用的 web.xml 文件会被解析，文件里面的每一个 Servlet，Filter和 Listener 类被创建一个对象实例，也被保存在服务器内存中。当 servlet 容器关闭时，将会卸载所有的 web 应用，同时 ServletContext，Servlet，Filter，Listener类的实例均被当做垃圾进行回收

根据 java 文档说明，ServletContext 类定义了一组使 servlet 类和 servlet 容器进行交互的方法，比如获取文件的 MIME 类型，转发请求或者生成日志文件。需要注意的是，当一个 web 应用在部署文件中声明为分布式时，web 应用的每一个虚拟机都有一个上下文对象，因而此时上下文对象不能作为全局变量，我们可以使用数据库等外部资源来代替

ServletRequest 和 ServletResponse 从哪里开始他们的生命周期？

web 服务器监听来自特定端口(通常是80号端口)的 http 请求，当客户端(通常是用户通过浏览器)向服务器发送一个 http 请求后，web 服务器中的 Servlet 容器会创建 HttpServletRequest 和 HttpServletResponse 对象，并且通过已经创建的与客户端请求的 URL 地址相匹配的 Filter，Servlet 类中的方法来传递这些对象，所有这些都使用同一个线程

request 对象提供获取 HTTP 请求中所有内容的方法，比如 http 请求的头信息或者 body 信息。response 对象提供操作和发送 HTTP 响应的方法，比如设置响应的头信息或者 body 信息(通常是通过 JSP 页面中的 html 代码实现)。当一个 http 响应提交并结束后，request 和 response 对象会被标记为垃圾(多说一句，http 协议是一个无状态的协议，它不会记忆每次请求，一次请求响应结束后，下一次请求响应对于 http 协议来说是全新的)

如何管理 Session 对象，和 Cookie 有什么关系？

当用户第一次访问 web 应用时，或者第一次通过 request.getSession() 方法获取 HttpSession 对象时，servlet 容器会创建一个 session 对象，并给他一个独一无二的 ID(你可以通过 session.getID 来查看)，然后把 session 对象存储在服务器内存中。同时，在 http 响应中，web 容器会创建一个名为 JSESSIONID 的对象并把之前创建的 session 对象的独一无二的 ID 值作为这里 cookie 的值

根据 HTTP cookie 规范(正规的 web 浏览器和 web 服务器都必须遵守的规范)，随后的客户端请求需要带有该 cookie 对象，servlet 容器将从每次发送过来的 http 请求中根据名为 JSESSIONID 的cookie对象获取相应的 cookie 值，并且把该值与 web 服务器内存中的 HttpSession 值进行对比，通过这种方式来实现对同一会话的记忆

当超过默认的时间时，HttpSession 对象将失效，该默认时间通常是在 web.xml 文件中定义的， 通常为 30 分钟，也就是说，当客户端长达30分钟没有访问该 web 应用时，servlet 容器就会结束 session 的生存，随后的请求即使包含对应的 session 对象也不能访问到相同的 session 了，servlet 容器会创建一个新的 session 对象

同时，客户端中 session 对象的 cookie 值也有一个默认的生存时间，该时间也就是浏览器的运行时间。所以当客户端关闭浏览器(关闭所有的页面或标签)后，客户端的 session 就会被销毁。重新打开浏览器后，与之前服务器端 Session 相关的 cookie 值就不再发送了，而是重新开始
